## What Went Well

1. **Incremental detail assembly worked.** Stories 3.1→3.5 stacked sections in mockup order without rework — each story extended `ResortDetailContent` / `GettingThereSection` cleanly.
2. **Content mappers kept UI thin.** `detailSections.ts`, `parseBulletList.ts`, and `gatewayExperiences.ts` centralise frontmatter → view-model logic; components stay presentational.
3. **Phase 1 product decisions landed fast.** Removing Book chips (3.4) and deferring AI (3.6) avoided building features that conflict with launch scope — user-driven, docs updated same session.
4. **Analytics wired at point of use.** `resort_detail_viewed`, trail map, external link, and getting-there events instrumented in the components that own the interaction — ready for Story 5.4 verification.
5. **Shared-element transition held.** Card→detail hero uses the same `getResortCardImage()` source and transition name from Epic 2 — no card tap rework needed.
6. **CS → DS → CR rhythm continued.** Five stories shipped sequentially with approve outcomes; no patch stories required (unlike Epic 2 quiz podium).

---

## Challenges & Growth Areas

1. **Resort photography still missing.** All 20 resorts have empty `images.card` / `images.hero` — detail and list show gradient placeholders; `UnsplashAttribution` rarely renders. Launch gate 5.5 requires site-wide attribution — **content/editorial blocker**, not code.
2. **`last_reviewed` empty everywhere.** T2 season labels work from `seasonConfirmed`, but editorial last-reviewed line is hidden on all resorts until content team fills fields.
3. **Gateway data uneven.** Two resorts have <3 `nearby_daytrips` — gateway cards correctly hidden, but Getting There feels thinner; editorial gap or accept for Phase 1.
4. **Practical tips are app-wide, not per-resort.** Acceptable for Phase 1 (shared Japan-wide FAQ in i18n), but power users may expect resort-specific tips later.
5. **Highlight/lowlight icons deferred.** PRD mentions 16px icons; mockup-matched text-only columns shipped — polish pass before launch if DESIGN.md requires icons.
6. **Mid-epic scope realignment.** Original plan had 3.6 stub in Phase 1; deferral to Phase 3 required doc sweep (epics, PRD, tracking plan, onboarding). Lesson: lock monetisation/AI gates in planning before DS on dependent stories.
7. **Epic 3 code not yet committed post-3.1–3.5.** Initial repo commit covers Epic 1–2 only; Epic 3 implementation exists locally — commit before Epic 5 PRs.

---

## Key Insights

1. **`nearby_daytrips` doubles as gateway experiences.** Reusing existing frontmatter avoided a new schema for 3.5 — pragmatic CONTENT_MODEL interpretation; document in content guidelines for editors.
2. **IntersectionObserver for section views is lightweight.** `getting_there_viewed` fires once on visibility — pattern reusable for future scroll-depth analytics if needed.
3. **Informational-only transport is the right Phase 1 default.** Aligns with PRD §11.1 (no monetisation); booking chips can return in monetisation phase without UI rework if schema added later.
4. **Phase 3 AI bundle is cleaner than a stub.** Deferring 3.6 entirely avoids a dead-end floating button and orphaned `/api/chat` UX debt at launch.
5. **Detail page is now the product centrepiece.** Discovery (Epic 2) + detail (Epic 3) complete the core browse→research loop; Epic 5 adds trust/legal/analytics verification to ship.
