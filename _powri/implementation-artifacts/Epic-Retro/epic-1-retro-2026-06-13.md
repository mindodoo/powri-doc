## What Went Well

1. **Sequential CR → CS → DS workflow held.** Each story built cleanly on the previous one; no rework cascades.
2. **Content pipeline early.** Story 1.3 gave typed resort data before routing (1.4) needed it — SSG placeholders worked first try.
3. **Cross-story coordination.** Analytics consent z-index (1.4→1.5), CSP/SRI (1.5→1.6), middleware merge (1.6→1.7) were anticipated in story specs and landed without conflict.
4. **Design tokens first.** Theme A in 1.2 meant all subsequent UI used consistent magazine typography from day one.
5. **Security and privacy from launch.** Consent-gated analytics, strict CSP, and API scaffolds are in place before any user-facing AI or search features (Epic 4).

---

## Challenges & Growth Areas

1. **All resorts still draft.** Home shows "0 published resorts" — expected for foundation epic, but Epic 2 resort cards will look empty until editorial publishes at least a subset.
2. **Placeholder-heavy UX.** Home, Explore, Info, and resort detail pages are scaffold copy, not product screens. Stakeholder demos need framing: "this is the shell, Epic 2 is the experience."
3. **Next.js 16 middleware deprecation warning.** `"middleware" file convention is deprecated. Please use proxy instead."` — monitor Next.js migration path; no action required for Phase 1 but track before production hardening.
4. **In-memory rate limiter.** Story 1.6 defers Upstash/Redis for multi-instance deploys — acceptable for dev/single-instance Vercel, must revisit before scale.
5. **Nested `web/.git`.** Still present from create-next-app; repo root has no VCS. Initialize monorepo git at root and remove nested git before first real deploy.

---

## Key Insights

1. **Foundation epic ≠ user-visible product.** Epic 1 success is measured by build pipeline, security headers, and developer ergonomics — not resort discovery UX.
2. **Draft vs published split is intentional.** `getAllResorts()` filters published only; `getResortBySlugFromFiles()` serves SSG placeholders for all 20 slugs — correct for dev, but Home list will stay empty until content gate moves.
3. **i18n routing gap caught at demo.** Story 1.7 initially placed pages at app root; next-intl middleware rewrites to `/en` internally, requiring `app/[locale]/` — fixed before demo.
4. **i18n last was the right order.** Migrating strings after all pages existed meant one comprehensive `en.json` rather than incremental churn.
5. **Middleware composition pattern works.** API CORS/rate-limit + next-intl in a single file with path-based branching is clean and extensible.
